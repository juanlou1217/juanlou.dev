# 任务类型与文档要求

这份文档用于回答一个很实际的问题：

一个任务来了，到底该不该写 spec？如果要写，应该写到什么程度？

## 1. 任务类型

### `FEATURE`

适用于：

- 新增用户可感知能力
- 恢复一个曾经关闭或占位的能力
- 给现有主线能力增加明显新行为

最低要求：

- 新建 `FEATURE_<NAME>.md`
- 明确当前产品状态
- 明确复用检查
- 明确验收标准

### `BUG`

适用于：

- 修复已存在的错误行为
- 修复页面不可访问、路由错误、数据错误、构建错误

最低要求：

- 新建 `BUG_<NAME>.md`
- 写清楚复现方式
- 写清楚预期行为和实际行为

### `REFACTOR`

适用于：

- 结构整理
- 复用收敛
- 去重复
- 目录、layout、组件职责重组

最低要求：

- 新建 `REFACTOR_<NAME>.md`
- 写清楚为什么当前结构不再适合继续维护
- 写清楚哪些能力不应该被改坏

### `DOC`

适用于：

- 新增项目文档
- 调整 harness 文档
- 更新 AI 协作规则

可以不单独起完整 spec，但建议至少在 trace 里留原因和范围。

## 2. 什么时候可以不写完整 spec

以下情况可以不写完整 spec：

- 只改一个明显错字
- 只修一个极小的样式问题
- 只补一个非常明确的链接地址
- 只做纯文案调整，且不影响边界

但仍建议：

- 至少经过任务开始检查清单
- 在提交说明或 trace 中说明范围

## 3. 什么时候必须写 spec

以下情况默认必须写：

- 要恢复 `DISABLED` 能力
- 要扩展 `LIMITED` 能力
- 要新增公开路由
- 要新增新的通用组件或 layout
- 要改变真实数据源
- 要大范围重构首页、博客列表、文章详情、标签或 RSS

## 4. spec 完成后至少补什么

即使任务本身不大，spec 至少应回答：

- 当前产品状态是什么
- 本次为什么现在做
- 搜索并复用了哪些现有实现
- 明确不做什么
- 如何验证
