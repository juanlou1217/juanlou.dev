# 任务开始检查清单

这份清单用于约束 AI agent 或维护者在真正开始改代码前，先完成最小上下文确认。

目标不是增加流程负担，而是防止：

- 重复造轮子
- 误恢复已关闭功能
- 在错误的数据源上继续叠逻辑
- 没看清当前产品面就直接改代码

## 1. 产品面确认

- [ ] 这个任务涉及的页面或能力，当前是否属于对外开放能力？
- [ ] 如果它属于预留/关闭能力，这次是否有用户明确要求恢复？
- [ ] 如果没有明确要求，是否应保持 404、占位页或未启用状态？

参考：

- [Repo-Specific 协作规则](../canonical/REPO_SPECIFIC_RULES.md)
- [复用与结构契约](../contracts/REUSE_AND_STRUCTURE_CONTRACTS.md)

## 2. 复用检查

至少搜索一次以下目录，再决定是否新增实现：

- [ ] `components/ui/`
- [ ] `components/blog/`
- [ ] `components/homepage/`
- [ ] `layouts/`
- [ ] `lib/`
- [ ] `app/`

至少回答这 3 个问题：

- [ ] 是否已有可复用组件？
- [ ] 是否已有可复用 layout？
- [ ] 是否已有可复用 route / util / service？

## 3. 数据源确认

- [ ] 这次改动的数据应该来自真实内容，还是来自静态手写数据？
- [ ] 如果是标签、分类、RSS、文章元信息，是否优先使用了 MDX / Contentlayer / 构建产物？
- [ ] 是否避免了“手写一个看起来真实、实际会漂移的展示数组”？

## 4. 路由与公开性确认

- [ ] 这次改动会不会新增用户可访问入口？
- [ ] 如果新增入口，是否需要同步更新导航、sitemap、metadata 或 contract？
- [ ] 如果关闭入口，是否需要移出 sitemap 或显式返回 404 / redirect？

## 5. 文档同步确认

- [ ] 这次改动是否改变了 repo-specific 事实？
- [ ] 如果改变了，是否需要同步更新 `canonical/`？
- [ ] 如果改变了边界或复用规则，是否需要同步更新 `contracts/`？
- [ ] 如果改变了验证方式，是否需要同步更新 `verification/`？

## 6. 最小记录建议

开始实施前，建议在 spec 中至少补这 4 行：

- 当前产品状态是什么
- 搜过哪些现有实现
- 最终决定复用什么
- 明确不做什么
