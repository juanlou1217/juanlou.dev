# 博客项目搭建拆解

这份文档不是单纯的启动说明，而是给“准备写一篇这个博客是怎么搭起来的文章”的素材文档。

你可以把它理解为：

- 项目搭建过程的技术拆解
- 一篇博客搭建实录的提纲底稿
- 面向读者讲清楚技术取舍的讲稿

## 这篇博文适合怎么写

如果你准备发一篇介绍这个项目搭建过程的文章，建议不要写成“技术名词流水账”，而是围绕下面这条主线展开：

```text
为什么要自己做博客
  -> 我希望这个博客解决什么问题
  -> 我为什么没有选传统 CMS
  -> 我最终怎么把内容系统、页面系统和动态能力拼起来
  -> 做完以后，这个博客具备了什么能力，刻意没做什么
```

这样文章会比“我用了 Next.js、Tailwind、Prisma”更有可读性。

## 推荐文章结构

下面这个结构可以直接作为你的博文目录。

### 1. 为什么重新搭自己的博客

这一部分建议先讲动机，不急着讲技术：

- 我想有一个长期可控的内容阵地
- 我希望技术文章和生活随笔能放在同一个站点里
- 我不想把内容维护依赖在重型 CMS 上
- 我希望博客本身也是一个可以持续演进的工程项目

这能帮助读者理解：你做的是“个人内容系统”，不是模板换皮。

### 2. 我给这个博客定了什么边界

这部分很重要，因为它决定整个技术方案。

当前这个博客的主线很明确：

- 首页
- 博客列表
- 博客分类
- 标签页
- 文章详情
- 关于页
- RSS
- 评论
- 文章统计

同时你也刻意没有把它做成这些东西：

- 后台内容管理系统
- 用户系统
- 复杂项目展示平台
- 邮件营销系统
- 多语言国际化站点

这一段非常适合拿来写“做减法”的思路。

### 3. 为什么选择 Next.js 16 + React 19 + TypeScript

这里可以从工程视角来讲：

- Next.js 负责路由、静态页面、API Routes、SEO 输出
- React 负责组件层和交互层
- TypeScript 负责约束内容结构、服务接口和组件 props

你可以强调一点：

这个博客不是只需要“渲染文章”，它还需要：

- 分类和标签聚合
- 元数据和结构化数据
- 少量 API 能力
- 后续持续维护的清晰结构

所以 Next.js 的整合能力比单纯静态站生成器更适合你当前阶段。

### 4. 为什么内容层选 MDX + Contentlayer

这是这篇文章里最值得展开的一节。

你可以直接写出你的核心判断：

- 我希望内容跟着 Git 走
- 我希望 Markdown 写作体验保留
- 我希望文章元信息有类型约束
- 我希望标签、阅读时长、目录、搜索索引能自动生成

这个仓库里的内容链路是：

```text
data/blog/*.mdx
  -> Contentlayer 解析
  -> 生成类型和计算字段
  -> 生成标签统计与搜索索引
  -> 页面读取内容对象并渲染
```

可重点讲的几个细节：

- frontmatter 里明确约束了 `title`、`date`、`tags`、`category` 等字段
- `category` 当前只允许 `tech`、`life`、`essay`
- 构建时会生成 `readingTime`、`toc`、`slug`、`structuredData`
- 构建成功后会额外写出 `app/tag-data.json` 和 `public/search.json`

这能体现一个观点：**内容系统不是“存文章”，而是“为页面和功能提供结构化内容数据”**。

### 5. 页面是怎么组织起来的

这一节可以解释你为什么没有把所有页面逻辑都塞在 `app/` 里。

你现在的层次是：

- `app/`：路由入口、页面文件、API Routes
- `layouts/`：列表页和文章页骨架
- `components/`：页面部件和通用 UI
- `lib/`：内容函数、SEO、服务逻辑
- `data/`：文章、作者、站点配置

读者一看就会明白：

- 路由和 UI 结构是分开的
- 页面骨架和页面内容是分开的
- 内容文件和运行时代码是分开的

这一段很适合配一张目录结构图或者架构图。

### 6. 为什么这个博客不是纯静态站

这会是文章里一个很自然的转折点。

虽然大部分页面都来自静态内容，但你还是加了几类“轻动态”能力：

- 文章浏览量和互动统计
- GitHub 仓库信息
- Giscus 评论

其中最值得讲的是统计系统：

```text
文章页触发浏览或互动
  -> 前端组件或 Hook 发请求
  -> /api/stats
  -> Prisma
  -> PostgreSQL
```

这个实现很适合说明你的设计哲学：

- 内容页仍然以静态为主
- 真正需要实时性的功能再走 API
- 数据库只承担少量结构化数据，不侵入整个内容系统

### 7. GitHub 数据为什么单独做服务层

`/api/github` 背后并不是页面里直接 `fetch` GitHub API，而是单独放在服务层里处理。

这个设计的好处是：

- 页面组件不用关心 GitHub API 细节
- 有 Token 时可以走 GraphQL
- 没 Token 时还能降级到公共 REST API
- 后续要改缓存策略或字段结构，改动集中在服务层

如果你想让文章更偏工程一点，这一段可以单独展开。

### 8. 评论、搜索、RSS 为什么值得保留

这些能力虽然不是项目最核心的“炫技点”，但很能体现博客是否完整。

- 评论：用 Giscus，避免自建评论系统
- 搜索：构建时生成本地索引，避免引入外部搜索平台
- RSS：保留技术博客最经典的订阅方式

你可以把这一节写成“博客不是只有文章页，周边配套也决定体验”。

### 9. 为什么要做文档分层

这是这个仓库和很多个人博客项目不一样的地方。

你现在把文档拆成了两层：

- `docs/project/`：面向 GitHub 访客和普通读者
- `docs/harness/`：面向维护者和 AI coding agent

这背后的思路可以直接写进博文：

- 当项目越来越大时，代码本身不够表达“当前主线是什么”
- 需要有一层文档去明确哪些能力开放、哪些能力关闭、哪些目录应该复用
- 如果以后让 AI 或其他协作者接手，仓库需要先变成“可解释的”

这部分会让你的文章从“搭博客”上升到“怎么把个人项目做成长期可维护的工程”。

## 可直接写进文章的几个亮点

如果你想让文章更有辨识度，建议重点强调这几个点。

### 1. 不是上 CMS，而是把内容系统放回 Git

读者会很容易理解这个判断，因为它直连写作体验和长期维护成本。

### 2. 静态博客不等于没有动态能力

你这个项目就是一个典型例子：静态内容页为主，动态能力局部补上。

### 3. 仓库边界是主动收敛出来的

你没有继续扩展 `/projects`、复杂 `lab demos`、newsletter、i18n，这反而体现了产品判断。

### 4. 文档系统也属于项目的一部分

不是“代码写完就算结束”，而是把项目说明、结构说明、协作规则也补齐了。

## 文章里可以引用的代码入口

如果你写博文时想贴代码片段或截图，下面这些位置最值得引用：

- `contentlayer.config.ts`
  内容 schema、计算字段、标签统计、搜索索引生成
- `app/blog/[...slug]/page.tsx`
  文章详情页、布局选择、结构化数据输出
- `lib/content.ts`
  发布内容过滤、分类与标签查询
- `app/api/stats/route.ts`
  互动统计 API
- `lib/services/prisma.ts`
  Prisma 与 PostgreSQL 连接
- `app/api/github/route.ts`
  GitHub 仓库信息 API 入口
- `lib/services/github.ts`
  GraphQL / REST 双路径数据获取
- `data/siteMetadata.js`
  站点基本信息、评论、搜索、分析配置

## 你可以怎么收尾

一篇搭建文章最后不要只写“这个博客已经完成了”，更适合写成：

- 当前版本已经把博客主线跑通
- 后续还可以继续迭代视觉、内容和少量交互
- 但不会为了“功能更全”而牺牲主线清晰度

这样的收尾会比“后面还会继续更新”更具体，也更符合这个仓库现在的状态。
